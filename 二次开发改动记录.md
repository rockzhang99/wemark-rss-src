## 改动 002

**日期**：2026-07-14

**提示词**：
> #改动002 我在后台点着点着就会提示 http://localhost:3000/login?redirect=/tags&error=session_expired，我的token肯定没到期，请认真检查过期的原因，别无故就跳出去重新登录

**问题根因**：
1. 路由守卫（`web_ui/src/router/index.ts`）在**每次路由导航都会请求后端 `verifyToken()`**。只要该请求因网络抖动、后端瞬时 500、超时等失败，就会进 catch → 清除 token → 强制跳登录并带 `error=session_expired`。这是“无故跳出去”的核心 bug（URL 中的 query 正由路由守卫生成，http 拦截器跳登录不带 query）。
2. 后端 `core/auth.py` 中 `cfg.get("token_expire_minutes", 30)` 回退值仅 30 分钟；若 `config.yaml` 缺失该配置项即悄悄回退到 30 分钟（文档/示例配置应为 4320=3 天），导致“点着点着就真过期”。

**改动内容**：
- `web_ui/src/utils/auth.ts`：新增 `parseJwt()`（本地解码 JWT payload）与 `isTokenExpired()`（本地判断过期，留 10s 缓冲），无需请求后端。
- `web_ui/src/router/index.ts`：路由守卫改为**本地 `isTokenExpired()` 检查**，移除每次导航的 `verifyToken()` 后端请求；真正的 token 失效由 http 拦截器 401 兜底。新增 `isTokenExpired` 导入。
- `web_ui/src/api/http.ts`：401 响应/错误分支跳登录时统一带上 `query: { error: 'session_expired' }`，保持体验一致。
- `core/auth.py`：将 token 过期回退默认值由 `30` 改为 `4320`（与 `config.example.yaml`、文档一致，即 3 天），避免配置缺失时过短。

## 改动 004

**日期**：2026-07-14

**提示词**：
> data/doc目录下有全部文章.zip，但是导出时前台进度条不动，http://localhost:3000/export/records下没有任何记录

**问题根因**：
1. 导出“全部”公众号时 `mp_id` 为**空字符串 `''`**（`activeFeed.id` 为空）。前端 `ExportModal.vue` 的 `pollStatus` 里有 `if (!currentMpId) return;`，空字符串是 falsy，导致轮询每次直接 return，**进度条永远停在 0**。
2. 后端 `export/status`、`export/list` 对空 `mp_id` 不够健壮：`export/status` 用 `Query(...)` 强制必填；`export/list` 若拿到 `None` 会在 `os.path.join(safe_root, None)` 抛异常 → 返回 500 → 前端 catch → 记录列表为空。
3. Windows 下 `os.path.relpath` 产生反斜杠，拼进 `download_url` 会破坏 URL，导致后续下载失败。
4. `/export/status` 属新增路由，后端未重启时会 404，轮询进 catch 直接停掉，进度同样不动。

**改动内容**：
- `web_ui/src/components/ExportModal.vue`：
  - 轮询状态改用独立布尔标志 `polling` 控制，`currentMpId` 允许为空字符串（改为 `string | null`），修复 `mp_id=''`（导出全部）时进度条不动的问题。
  - `stopPolling` 同步置 `polling=false`。
- `apis/tools.py`：
  - `export/status` 参数改为 `Query("")` 并做 `mp_id = mp_id or ""` 归一化，兼容“导出全部”。
  - `export/list` 新增 `mp_id = mp_id or ""` 归一化，避免 `None` 导致异常，空值时正常遍历全部 zip。
  - `export/list` 返回的 `path` 与 `download_url` 统一将路径分隔符转为正斜杠，修复 Windows 反斜杠破坏下载链接的问题。

**说明**：修改后端 Python 代码后**必须重启后端进程**（`python main.py`），否则新增/修改的 `/export/status`、`/export/list` 逻辑不生效（旧进程会 404 或走旧逻辑）。

## 改动 005

**日期**：2026-07-14

**提示词**：
> 导出进度提示了10000%，实际导出记录里是空的，进度是正常了

**问题根因**：
1. **进度 10000%**：`tools/mdtools/export.py` 里 `total`（进度分母）原来用 `page_count * page_size` 估算（前端默认 10×10=100）。但 `process_articles` 实际处理的文章数远不止这个数——尤其导出“全部”时 `mp_id=''` 不过滤，文章库有数万篇；以及“已选文章”导出走 `doc_id` 分支根本不限页数。于是 `done` 远超 `total`，`int(done/total*100)` 直接爆成几千上万百分比。
2. **记录为空**：进度算错只是显示问题，记录空更可能是——`page_count*page_size` 只导出了前 100 篇，而 PDF 生成很慢（每篇起浏览器），任务还在跑或中断，zip 未生成；加上进度误报“完成”去查记录自然为空。同时“全部”导出只导出前 100 篇，与文件名“全部文章.zip”名不副实。

**改动内容**：
- `tools/mdtools/export.py`（`export_md_to_doc`）：
  - 进度分母 `total` 改为用**真实文章数**（`_q.count()`）计算：当 `page_count>0` 时取 `min(page_count*page_size, real_total)`，否则取 `real_total`，保证 `done` 永远不超过 `total`。
  - 新增 `effective_page_count`：**导出“全部”文章（`mp_id` 为空且无 `doc_id`）时不受页数限制**，导出所有文章，让“全部文章.zip”名副其实。该值同时用于 `total` 计算与 `process_articles` 调用。
- `apis/tools.py`（`_make_progress_callback`）：
  - 进度百分比增加上限裁剪 `min(100, int(done/total*100))`，杜绝任何情况下进度超过 100%。

**说明**：本次修改了后端 Python，需**重启后端进程**（`python main.py`）才能生效。重启后重新导出“全部”：进度会从 0 平滑走到 100%，任务真正完成后 `data/docs/全部文章.zip` 生成，再到“导出记录”页刷新即可看到。若文章量很大（数万篇），PDF 导出较慢请耐心等待任务结束（状态变为 done）再查记录。

## 问题修复（非 #改动 标记，运行环境/配置类）

### 修复 1：任务队列 WebSocket 连接 1006 失败

- **现象**：`TaskQueueView.vue` 控制台报 `WebSocket connection to 'ws://localhost:3000/api/v1/wx/task-queue/ws' failed: WebSocket is closed before the connection is established.`，错误码 1006。
- **根因**：前端在 3000 端口（vite dev server）访问，WebSocket 请求 `ws://localhost:3000/api/...` 经 vite 代理转发到后端 8001，但 `web_ui/vite.config.ts` 中 `/api` 代理**未开启 `ws: true`**，vite 不处理 WebSocket 升级请求，连接被异常关闭。
- **修复**：`web_ui/vite.config.ts` 的 `/api` 代理增加 `ws: true`。
- **生效**：重启前端 dev server（`npm run dev`）。

### 修复 2：HTTP 请求带 /export 前缀导致 404

- **现象**：控制台报 `GET http://localhost:3000/export/api/v1/wx/user 404`、`GET http://localhost:3000/export/api/v1/wx/tools/export/list?mp_id= 404`。注意 WebSocket 不带 `/export`（其 URL 运行时用 `window.location.host` 拼接，不走 axios baseURL）。
- **根因**：`web_ui/src/api/http.ts` 中 `baseURL: (import.meta.env.VITE_API_BASE_URL || '') + 'api/v1/'`。当 `VITE_API_BASE_URL` 被设为 `/export/`（常见于 shell 环境变量、`.env.local` 或临时配置污染 dev server）时，所有 HTTP 请求都带上 `/export` 前缀；而后端路由（`API_BASE = /api/v1/wx`）没有 `/export` 前缀，于是 404。
- **修复**：`web_ui/src/api/http.ts` 改为 dev 模式（`import.meta.env.DEV`）强制使用相对路径 `api/v1/`（由 vite 代理转发到后端），不受 `VITE_API_BASE_URL` 污染；生产模式仍保留 `VITE_API_BASE_URL` 以支持子路径部署（后端无子路径前缀时应设为空或 `/`）。
- **生效**：重启前端 dev server（`npm run dev`）。
- **补充**：若以构建版（`npm run build` 后由 `vite preview` 或静态服务/反向代理在 3000 端口提供）访问，需确保构建时 `VITE_API_BASE_URL` 为空或 `/`，并重新 `npm run build`；否则仍会带错误前缀。后端路由当前无 `/export` 前缀，前端不应使用 `/export` 作为 API 前缀。

#### 修复 2 二次排查与加固（2026-07-14）

- **现象（用户反馈）**：在 `npm run dev` 启动的前端（端口 3001）点击 `http://localhost:3001/export/records`，控制台报 `auth.ts:96 GET http://localhost:3001/export/api/v1/wx/user 404`，`App.vue:150 获取用户信息失败`。
- **关键排查（实测，非推测）**：直接 `curl http://localhost:3001/src/api/http.ts` 抓取 3001 dev server **正在提供**的模块源码，确认：
  - 第 1 行 `import.meta.env` 中 `VITE_API_BASE_URL: ""`、`DEV: true`；
  - dev 分支 `const apiBase = import.meta.env.DEV ? "" : ...`，故 `baseURL = "api/v1/"`，**新代码根本不会产生 `/export` 前缀**。
  - 结论：用户浏览器里那个标签页跑的是**改代码之前加载的旧 JS**（旧 axios 单例 baseURL 仍带 `/export`，是当初 `VITE_API_BASE_URL=/export` 时加载并缓存的）。dev server 重启到 3001 后新代码已就绪，但旧标签页未刷新，仍在发 `/export/api/v1/wx/user`。
- **加固修复**：`web_ui/src/api/http.ts` dev 分支由相对路径 `api/v1/` 改为**绝对路径** `/api/v1/`（`apiBase = import.meta.env.DEV ? '/' : ...`）。原因：原相对路径 `api/v1/` 在页面停在 `/export/records` 等子路由时，会被浏览器按 `document.baseURI` 解析成 `/export/records/api/v1/...` 而 404；改为带前导斜杠的绝对路径后，无论当前页面路径如何，请求始终是 `/api/v1/...`，被 vite `/api` 代理正确转发到后端 8001。
- **生效动作（给用户）**：
  1. **硬刷新浏览器**（`Ctrl+Shift+R`）或关闭旧标签页重新打开 `http://localhost:3001/export/records`——这是必须的，旧标签页缓存的旧 bundle 不会自动更新 axios 单例。

- **环境提醒**：用户环境曾把 `VITE_API_BASE_URL` 设为 `/export`（旧标签页即彼时加载）。当前 3001 显示已为空，但应确认 `.env`/`.env.local`/shell 中无常驻该变量，否则将来 `npm run build` 部署时生产模式又会带上 `/export` 而 404。检查：`cat web_ui/.env web_ui/.env.local`、`echo %VITE_API_BASE_URL%`。

**日期**：2026-07-16

**问题**：

**原因**：

- 在 `Db.init()` 中调用 `self.ensure_subscription_table()`，使其随数据库连接初始化自动执行（无论是否跑 init_sys，启动即建表）。

- `python -m py_compile core/db.py` 通过，无 lint 错误。

**设计决策 / 说明**：
- 保留原 `init_sys` 的 `create_all` 路径不变，新增兜底仅针对此新增表；不影响其他表结构。

1. **为什么放弃云部署代理池**：微信主要封禁数据中心 IP（阿里云 ECS 等），住宅宽带 IP 基本不在封禁逻辑内。安装包分发 → 客户住宅 IP → 天然规避封禁，故此前为"隐藏数据中心 IP"而建的 Cloudflare 代理池（`proxy_pool.py`、`worker.js`）出发点消失，已彻底清除，代码天然支持 `proxy.enabled: False` 直连住宅 IP。
2. **为什么不用 Electron**：本工程核心是 Python（FastAPI + Playwright 抓取 + 反爬），Electron 只能打包 Web 前端、跑不了 Python，后端仍需 PyInstaller 各平台各打一次。且 Electron 自带一个 Chromium，工程 Playwright 已需一个 Chromium，上 Electron 会变成两个 Chromium，安装包 +150MB，与"尽可能小"冲突。除非需要原生桌面窗口/托盘/自动更新，否则不必引入。
3. **最终方案**：PyInstaller 按平台各构建一次（Windows 出 exe 目录、Mac 出 .app 目录），前端静态已在 `static/`，运行时用户访问 `localhost`。Chromium 不打进安装包，首次启动自动 `playwright install chromium`（安装包 ~80MB，首次联网下载 ~150MB）。

**依赖与体积盘点**：
- **Redis**：`tools/redis_server.py` 是纯 Python 实现的 RESP 服务端，`core/redis_client.py` 未配 `redis.url` 时优雅降级，无需打包任何 Redis 二进制，跨平台无坑。
- **selenium / webdriver-manager**：`requirements.txt` 中有但全代码 0 引用，打包 `exclude`。
- **psycopg2 / PyMySQL**：全代码 0 引用，打包 `exclude`。
- **torch / onnx / opencv**：完全没有，Python 侧体积可控。
- **Chromium 浏览器二进制**：每个 ~150MB，不在 pip 包内，外置首次下载（体积大头）。
- **前端静态**：已构建在 `static/`，运行时不需要 Node；`qtserver`（Node MQTT）为可选，可排除。
- 估算：PyInstaller + UPX 压缩后后端 + 依赖 + 前端约 **50–80MB**。

## 改动 036

**日期**：2026-07-17

**需求 / 现象**：
> 在改动 035（放弃云部署、改 PyInstaller 本地安装包）的基础上，重新调整整体设计为「混合架构」：本地可执行程序里**只有和微信的交互接口**跑在本地住宅 IP；另上传一个服务端到阿里云，阿里云上**不执行任何微信接口**；本地程序下载好文章后把数据保存到服务端。部署模式选 **A（多租户 SaaS：一台云服务器服务所有客户，每客户装轻量 agent）**，且**云端前端是纯 RSS 服务**（不提供订阅管理 UI，管理 UI 留在本地 agent 的 localhost）。

**可行性结论**：可行。核心安全性前提——阿里云数据中心 IP 永远不直接调微信 → 微信从来看不到该 IP → 不会被封。该前提靠「云端构建不打包 `driver/`/`core/wx/` 微信驱动代码」从物理上保证。

**设计要点（详见 `docs/混合架构_本地代理与云端RSS设计.md`）**：

1. **职责切分**：
   - 本地 Agent（住宅 IP，PyInstaller 打包）：微信授权、文章抓取（Playwright + `driver/` + `core/wx/`）、本地定时调度、管理 UI（localhost）、上传器（增量推送文章/Feed 到云端）。微信凭证**只留本地，不上传**。
   - 云端 Server（阿里云数据中心 IP，纯 RSS）：接收上传（AK 鉴权 + 按租户落库）、存储 `Feed`/`Article`、输出 RSS（只读、token 保护）、健康检查。**不 import 微信驱动、不提供授权、不提供管理 UI**。
2. **通信协议**：复用现有 AccessKey（AK/SK）机制；`POST /api/v1/wx/agent/feeds` 与 `POST /api/v1/wx/agent/articles`（批量，文章体直接复用 `Article.to_dict()`，云端用 `DB.add_article()` upsert 去重）；写入前强制打 `tenant_id`；必须 HTTPS。
3. **多租户隔离（模式 A 关键）**：`Feed`/`Article` 增加 `tenant_id`（复用改动 021 的 `owner` 范式）；`core/db.py` 增加 `ensure_tenant_columns()`；`apis/rss.py` **恢复 `get_current_user` 鉴权并按 `tenant_id` 过滤**（现有 RSS 端点鉴权被注释、对外敞开，上线前必须修复）。
4. **「立即刷新」与调度不污染云端**：刷新/调度均在本地 Agent 执行，云端零微信调用。
5. **构建拆分**：在改动 035 的 `build/` 基础上拆 `pyinstaller_agent.spec`（含微信驱动+管理 UI+Chromium 外置）与 `pyinstaller_cloud.spec`（exclude `driver`/`core.wx`/`playwright`/管理 UI，无 Chromium 下载）；云端加运行时守卫——若 `driver` 被 import 立即退出。
6. **配置**：`config.yaml` 新增 `deploy.role: agent|cloud`；Agent 侧 `upload.enabled/server/ak/sk`；Cloud 侧 `wechat.enabled: false`。

**风险 / 注意**：
- 多租户隔离是最大工作量，须同时锁死「写入打 tenant_id」与「RSS 读取按 tenant_id 过滤」，任一边漏了都会跨客户泄漏。
- 文章 `content` 内图片为微信 CDN 直链，RSS 阅读器直拉微信 CDN，云端 IP 不接触微信，安全。
- Agent 离线 = RSS 停滞（与纯本地方案一致）。

## 改动 036 - 阶段 4 落地（2026-07-17）

**目标**：联调与安全复核，把混合架构从「能跑」推进到「生产可用」——堵住跨租户越权、补全 RSS token 订阅可用性、给出 HTTPS 与多租户压测手段、确认 AK 轮换机制完备。

**已实施内容**（安全复核 + 配套文件）：

1. **修复 RSS 文章正文跨租户越权（中危漏洞）**：原 `apis/rss.py::get_rss_feed`（`/rss/content/{content_id}`）无认证、无租户隔离，云端多租户下任何知道 `article.id` 的人可读取他人文章正文。阶段4改为：加 `Depends(_rss_tenant_dep)`（云端模式强制认证）；云端且非 admin 时查 `Article.tenant_id` 比对当前租户，不匹配返回 404；本地 Agent 模式（`tenant_id=None`）行为不变，向后兼容。

2. **RSS token 租户绑定补全（可用性 + 隔离）**：原 `_rss_tenant_dep` 只支持 `Authorization` 头，RSS 阅读器（Feedly 等）无法在 URL 里传 token。阶段4支持 `?token=` 查询参数（自动识别 `AK-SK xxx:yyy` 与 `Bearer <jwt>`），并复用与 header 完全相同的租户解析与 admin 豁免逻辑——A 租户 token 看不到 B 租户数据。订阅地址示例：`https://rss.your-domain.com/rss?token=AK-SK%20WKxxx:SKxxx`。

3. **AK 轮换机制确认完备（无需改代码）**：`core/auth.py` + `apis/auth.py` 已提供完整 AK 生命周期——`/ak/create`（可设 `expires_in_days`、权限）、`/ak/list`、`/ak/{id}` PUT、`/ak/{id}/deactivate`、`/ak/{id}` DELETE。轮换即「新建 AK → 更新 Agent 端 `upload.ak/sk` → 停用旧 AK」。SecretKey 仅创建时明文返回一次，库内存 `sha256` 哈希。

4. **HTTPS 模板**：新增 `script/nginx.cloud.conf`——SSL 终止 + 按 IP 限流（`/api` 20r/s、`/rss`、`/feed` 10r/s）+ 安全响应头；后端只跑 `http://127.0.0.1:8001`，HTTPS 统一由反代处理。

5. **多租户压测脚本**：新增 `script/bench_cloud.py`——`requests` + 线程池，对 `/rss`（读）与 `/api/v1/wx/agent/articles`（写）并发压测，支持多对 `--ak/--sk` 模拟多租户，输出 QPS / p50 / p95 / p99 / 状态码分布。
