# 多用户数据隔离设计方案（订阅 / 标签 / 消息任务）

> 关联文档：`docs/角色权限与菜单可见性设计.md`
> 本文落地「真正的多用户隔离，普通用户可自建订阅/标签」。

## 1. 现状问题

上一阶段只解决了**菜单/路由入口**的按角色可见性，数据层仍是全局共享：

- `feeds`（订阅/公众号）、`tags`（标签）、`message_tasks`（消息任务）均无 `owner` 字段；
- 列表接口不分用户，`session.query(Feed)...` 返回全量数据；
- 任意登录用户（含普通用户）都能看到/操作其他用户的订阅、标签、任务。

## 2. 隔离模型决策

采用**「全局目录 + 订阅关系表」**方案，而非给每条数据复制多份：

- **公众号(Feed) 作为全局目录**：同一公众号只存一份，文章(Article)挂在全局 Feed 上，订阅者共享同一份抓取内容，避免重复抓取与存储。
- **新增 `subscriptions` 表**：记录 `user_id ↔ feed_id` 的订阅关系，实现「同一公众号可被多个用户各自订阅、各自启停」。
- **`tags` / `message_tasks` 增加 `owner` 字段**：直接归属到用户。
- **`filter_rules` 保持全局**：它是按公众号的内容清洗规则，本质是共享配置，不属于某用户私有数据（本期不做隔离，后续如需可按 `owner` 扩展）。

### 角色行为

| 操作 | 管理员 admin | 普通用户 user / editor |
|---|---|---|
| 订阅列表 | 看到全部公众号 | 仅看到自己订阅的 |
| 添加公众号 | 新建/更新全局目录并订阅 | 复用已有全局目录，仅建立自己的订阅 |
| 删除（取消订阅） | 全局删除公众号及其全部订阅 | 仅取消自己的订阅；无订阅者后清理孤立目录 |
| 标签 / 消息任务 | 全部 | 仅自己的 |

## 3. 数据模型变更

### 3.1 新增 `core/models/subscription.py`
```python
class Subscription(Base):
    __tablename__ = 'subscriptions'
    id = Column(String(255), primary_key=True, default=lambda: str(uuid.uuid4()))
    user_id = Column(String(255), nullable=False, index=True)  # 对应 username
    feed_id = Column(String(255), nullable=False, index=True)  # 对应 feeds.id
    status  = Column(Integer, default=1)                       # 用户维度启停
    created_at / updated_at
```
已在 `core/models/__init__.py` 注册，随 `Base.metadata.create_all` 自动建表。

### 3.2 `tags` / `message_tasks` 增加 `owner`
```python
owner = Column(String(255), nullable=True, index=True)
```

### 3.3 存量数据迁移 `core/db.py`
新增 `ensure_isolation_columns()`，在 `Db.init()` 中调用（与 `ensure_article_columns` 并列）：
- 对已有 `tags` / `message_tasks` 表 `ALTER TABLE ... ADD COLUMN owner VARCHAR(255)`（`create_all` 不会给旧表加列，必须显式迁移）；
- 历史数据 `UPDATE ... SET owner='admin' WHERE owner IS NULL`（初始管理员为 `admin`）；
- `subscriptions` 为新表，由 `create_all` 在建库时自动创建，无需迁移。

## 4. 接口变更

### 4.1 `apis/mps.py`（订阅）
- `GET /mps`：非管理员 `JOIN subscriptions` 仅返回自己订阅的公众号；管理员返回全部。
- `POST /mps`：订阅语义。全局目录已存在则复用并仅建订阅关系；不存在则新建 Feed 再订阅。普通用户重复订阅**不覆盖**全局展示信息（仅管理员可改）。
- `DELETE /mps/{id}`：管理员=全局删除公众号+全部订阅；普通用户=仅取消自己订阅，无订阅者后清理孤立目录。
- `PUT /mps/{id}`：普通用户仅维护自己订阅维度的 `status`；管理员维护全局公众号信息。
- `GET /mps/{id}`：补回鉴权，非管理员无订阅关系返回 403。
- `GET /mps/update/{id}`（同步）：非管理员无订阅关系返回 403。

### 4.2 `apis/tags.py`（标签）
- `GET /tags`：非管理员按 `owner` 过滤。
- `POST /tags`：写入 `owner=当前用户`。
- `GET/PUT/DELETE /tags/{id}`：非管理员校验 `owner`，越权返回 403。

### 4.3 `apis/message_task.py`（消息任务）
- `GET /message_tasks`：非管理员按 `owner` 过滤。
- `POST /message_tasks`：写入 `owner=当前用户`。
- `GET/PUT/DELETE /{id}`、`/{id}/run`、`/message/test/{id}`：非管理员校验 `owner`，越权返回 403。

> 消息任务创建时的「公众号选择器」调用 `GET /mps`，因该接口已按订阅过滤，前端自动只展示当前用户订阅的公众号，天然隔离。

## 5. 前端影响

- 后端过滤后，前端无需改动即可只看到自己的数据。
- `DELETE /mps/{id}` 语义由「删除公众号」变为「取消订阅」，前端刷新列表后即不再展示；按钮文案建议改为「取消订阅」（可选优化，不影响功能）。
- `rss.py` 公开 RSS 输出接口保持全局（RSS 阅读器凭 token 消费，符合预期），未做隔离。

## 6. 验证清单

1. `python -m py_compile` 全部改动文件通过。
2. 启动后数据库自动：新建 `subscriptions` 表；`tags`/`message_tasks` 增加 `owner` 列；历史数据 `owner=admin`。
3. 管理员登录：订阅管理/标签/消息任务看到全部数据。
4. 新建普通用户登录：
   - 订阅管理仅显示自己订阅的公众号；
   - 添加公众号→建立自己的订阅，不影响他人；
   - 删除→仅取消自己订阅；
   - 标签/消息任务仅显示自己创建的；
   - 直接 `GET /mps/{他人feed_id}` 或操作他人标签/任务 → 403。
5. 普通用户之间数据互不可见。

## 7. 后续可扩展

- 普通用户「自建订阅」已支持；如需限制可订阅的公众号范围，可在 `POST /mps` 增加白名单。
- `filter_rules` 如需按用户隔离，按本文 `owner` 模式扩展即可。
- 订阅维度的「启用/停用」目前仅控制列表展示；若需联动停止抓取，可在同步任务中读取 `subscriptions.status`。
