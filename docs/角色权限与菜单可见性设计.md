# 角色权限与菜单可见性设计

> 目标：让「角色」真正对应「功能菜单」。例如普通用户登录后，看不到系统管理类菜单（用户管理、级联、任务队列等），只能看到自己该用的功能。
> 本文档先分析现状，再给出完整设计方案（权限词表、角色预设、菜单可见性矩阵、前后端改造点）。后续可按需迭代。

---

## 1. 现状分析（非常重要：当前是「半成品」）

### 1.1 已具备的"骨架"

| 位置 | 内容 | 说明 |
|------|------|------|
| `core/models/user.py` | `role`（admin/editor/user）、`permissions`（JSON 文本）、`is_active` | 数据模型已支持角色 + 权限列表 |
| `apis/user.py` | 用户增删改查、重置密码 | 仅对 **`role == "admin"`** 做校验 |
| `core/auth.py` | `get_current_user_or_ak` 返回 `{username, role, permissions, ...}` | permissions 被返回，但**几乎没被强制校验** |
| `web_ui/src/router/index.ts` | 每个路由定义了 `meta.permissions` | 词表如 `wechat:manage` / `config:view` / `message_task:view` / `admin` |
| `web_ui/src/components/Layout/Navbar.vue` | `menuItems` 写死的 14 个菜单 | **没有任何角色/权限过滤，所有人看到全部菜单** |
| `web_ui/src/views/UserManagement.vue` | 角色下拉（管理员/编辑/普通用户）+ 权限勾选 | 权限选项是 `read/write/delete/admin`（见下方"不一致"） |

### 1.2 关键问题（为什么现在"角色不生效"）

1. **导航守卫不校验权限**：`router.beforeEach` 只检查 token 是否有效，**完全没用** `meta.permissions`。任何已登录用户手动输入 URL 都能进任意页面（含用户管理、系统信息）。
2. **菜单不按权限过滤**：`Navbar.vue` 的 `menuItems` 是静态数组，不会根据当前用户角色/权限隐藏菜单。
3. **前端拿不到自己的权限**：`api/user.ts` 的 `getUserInfo()` 返回 `role` 和 `is_active`，但 **`UserInfo` 接口没有 `permissions` 字段**，登录后前端根本不知道当前用户有哪些权限。也没有全局用户状态（store）保存角色/权限。
4. **后端不强制权限**：除用户管理接口校验 `role=="admin"` 外，其余接口的 `permissions` 形同虚设，从不参与鉴权。
5. **权限词表不一致（最坑）**：
   - 用户管理界面里勾选的权限是：`read` / `write` / `delete` / `admin`
   - 路由 `meta.permissions` 用的是：`wechat:manage` / `config:view` / `message_task:view` / `message_task:edit` / `tag:view` / `tag:edit` / `admin`
   - **两套词表对不上**，即使后端开始校验，勾了 `read` 也匹配不到 `config:view`，等于没用。

> 结论：数据模型和 UI 脚手架都在，但"权限真正落地到菜单/路由/接口"这一层缺失，且词表冲突。本设计就是补齐这一层并统一词表。

---

## 2. 统一权限词表（Permission Catalog）

以 `router/index.ts` 里已有的 `meta.permissions` 为准（它更贴合业务），并补全"订阅管理"等无权限页面。建议标准词表：

| 权限标识 | 含义 | 关联菜单/路由 |
|----------|------|---------------|
| `subscription:view` | 查看/管理订阅（文章列表） | 订阅管理 `/` |
| `wechat:manage` | 微信授权、过滤规则、公众号管理 | 授权管理、过滤规则、公众号管理(`wechat/mp`) |
| `tag:view` | 查看标签 | 标签管理 |
| `tag:edit` | 新建/编辑标签 | 标签管理(编辑) |
| `message_task:view` | 查看消息任务 | 消息任务 |
| `message_task:edit` | 新建/编辑消息任务 | 消息任务(编辑) |
| `config:view` | 查看配置、导出记录 | 配置信息、导出记录 |
| `admin` | 系统级管理（含用户管理） | 任务队列、公众号状态、级联管理、Access Key、用户管理、异常统计、系统信息 |

> 约定：路由 `meta.permissions` 为空（如 `订阅管理`）表示**所有已登录用户都可见**。

---

## 3. 角色预设（Role Presets）

采用"**角色给默认权限集 + 单用户可覆盖**"的模型（用户管理界面已有"不选择权限则用角色默认权限"的提示，本设计落地它）。

### 3.1 默认权限矩阵

| 权限 \ 角色 | 管理员 `admin` | 编辑 `editor` | 普通用户 `user` |
|-------------|:---:|:---:|:---:|
| `subscription:view` | ✅ | ✅ | ✅ |
| `wechat:manage` | ✅ | ✅ | ✅ |
| `tag:view` | ✅ | ✅ | ✅ |
| `tag:edit` | ✅ | ✅ | ⬜ |
| `message_task:view` | ✅ | ✅ | ✅ |
| `message_task:edit` | ✅ | ✅ | ⬜ |
| `config:view` | ✅ | ✅ | ✅ |
| `admin` | ✅ | ⬜ | ⬜ |

> 说明（可按业务调整）：
> - **普通用户**：只能看、不能用（不能建标签/任务），且**完全看不到系统管理类菜单**。
> - **编辑**：可日常运营（授权、标签、消息任务、配置查看），但**不能进系统管理**。
> - **管理员**：全部权限。
> - 单用户 `permissions` 字段若非空，则**覆盖**角色默认集（取并集或精确集，见 5.2）。

### 3.2 菜单可见性矩阵（用户最终看到的）

| 菜单 | 路由 | 所需权限 | 管理员 | 编辑 | 普通用户 |
|------|------|----------|:---:|:---:|:---:|
| 订阅管理 | `/` | （无） | ✅ | ✅ | ✅ |
| 授权管理 | `/wechat-status` | `wechat:manage` | ✅ | ✅ | ✅ |
| 导出记录 | `/export/records` | `config:view` | ✅ | ✅ | ✅ |
| 标签管理 | `/tags` | `tag:view` | ✅ | ✅ | ✅ |
| 消息任务 | `/message-tasks` | `message_task:view` | ✅ | ✅ | ✅ |
| 过滤规则 | `/filter-rules` | `wechat:manage` | ✅ | ✅ | ✅ |
| 配置信息 | `/configs` | `config:view` | ✅ | ✅ | ✅ |
| 任务队列 | `/task-queue` | `admin` | ✅ | ⬜ | ⬜ |
| 公众号状态 | `/cascade/feed-status` | `admin` | ✅ | ⬜ | ⬜ |
| 级联管理 | `/cascade` | `admin` | ✅ | ⬜ | ⬜ |
| Access Key | `/access-keys` | `admin` | ✅ | ⬜ | ⬜ |
| 用户管理 | `/users` | `admin` | ✅ | ⬜ | ⬜ |
| 异常统计 | `/env-exception` | `admin` | ✅ | ⬜ | ⬜ |
| 系统信息 | `/sys-info` | `admin` | ✅ | ⬜ | ⬜ |

> 可见性判定规则：**菜单对应路由的 `meta.permissions` 为空 → 所有人可见；非空 → 当前用户权限集包含其中至少一个即可见**。

## 4. 前端改造点（让菜单真正按角色隐藏）

### 4.1 全局用户状态（store / composable）
- 新增 `web_ui/src/store/user.ts`（或 composable）：登录成功后调用 `getUserInfo()`，把 `role` + `permissions` 存全局（pinia / reactive + provide）。
- `api/user.ts` 的 `UserInfo` 接口**补上 `permissions: string[]`**；`getUserInfo` 改为请求 `/user`（注意：当前 `api/user.ts` 用的是 `/wx/user`，需与后端 `apis/user.py` 的路由前缀对齐，见第 6 节）。

### 4.2 Navbar 按权限过滤菜单
- 给 `Navbar.vue` 的 `menuItems` 每一项加 `permission?` 字段（与路由 `meta.permissions` 对应）。
- 渲染时 `v-if` 过滤：`!item.permission || userPermissions.includes(item.permission)`。
- 桌面端横向菜单、移动端抽屉菜单、目录锚点同步过滤。

### 4.3 路由守卫强制校验（防 URL 直闯）
- `router.beforeEach` 在 token 有效后，增加：
  ```ts
  const required = to.meta.permissions as string[] | undefined
  if (required && required.length && !required.some(p => userPermissions.includes(p))) {
    return next({ path: '/', query: { error: 'no_permission' } })
  }
  ```
- 无权限访问时跳回首页并提示（可加一个 403 提示态）。

### 4.4 用户管理界面修正词表
- `UserManagement.vue` 的 `permissionOptions` 改为与第 2 节一致：`subscription:view` / `wechat:manage` / `tag:view` / `tag:edit` / `message_task:view` / `message_task:edit` / `config:view` / `admin`（中文 label 友好显示）。
- `checkPermission()` 改为：非 admin 直接 `next` 重定向，而不是只弹个提示还让页面渲染。

## 5. 后端改造点（让权限真正落地）

### 5.1 角色 → 默认权限映射
- 在 `core/auth.py` 或新建 `core/rbac.py` 定义：
  ```python
  ROLE_PERMISSIONS = {
      "admin":  ["subscription:view","wechat:manage","tag:view","tag:edit",
                 "message_task:view","message_task:edit","config:view","admin"],
      "editor": ["subscription:view","wechat:manage","tag:view","tag:edit",
                 "message_task:view","message_task:edit","config:view"],
      "user":   ["subscription:view","wechat:manage","tag:view",
                 "message_task:view","config:view"],
  }
  ```
- 计算"用户有效权限"：`user.permissions`（非空则以此为准，建议取并集避免误锁死）否则用 `ROLE_PERMISSIONS[role]`。

### 5.2 提供"有效权限"给前端
- `get_current_user_or_ak` 返回的 `permissions` 改为**计算后的有效权限集**（而不是原始存储串），前端无需再算角色映射。

### 5.3 接口级鉴权（可选但建议）
- 新增依赖 `require_permissions(*perms)`：当前用户有效权限需包含其一，否则 403。
- 逐步给敏感接口（用户管理、级联、配置写、Access Key 写）挂上，前端菜单隐藏只是"体验层"，后端校验才是"安全层"。

### 5.4 用户管理写接口
- `apis/user.py` 的 `add_user` / `update_user_by_id` 已支持写 `role` 和 `permissions`，**保持**；只需确保前端传的是统一词表即可。

## 6. 接口路径对齐（已核对：实际已对齐，无需改动）

> 落地后核对结论：前后端前缀**已经一致**，无需修改。
> - 前端 `api/user.ts` 调 `/wx/user`，而 `http.ts` 的 `baseURL` 为 `/api/v1/`，拼出实际请求 `/api/v1/wx/user`。
> - 后端 `apis/user.py` 路由前缀为 `/user`，挂在 `api_router(prefix=API_BASE)` 下，而 `API_BASE = /api/v1/wx`，拼出 `/api/v1/wx/user`，二者吻合。

真正缺失的是：**登录后前端没有拉取用户信息写入全局状态**。本次已在 `Login.vue` 登录成功后调用 `useUserStore().load()` 拉取并存储角色/权限，解决了"前端拿不到自身权限"的问题。

---

## 7. 实施步骤建议（落地顺序）

1. **统一词表 + 后端映射**：定义 `ROLE_PERMISSIONS`，让 `get_current_user_or_ak` 返回有效权限；对齐 `/user` 与 `/wx/user` 前缀。
2. **前端拿到权限**：`UserInfo` 补 `permissions`；登录后写入全局 store。
3. **Navbar 过滤**：菜单按权限显隐（用户立刻感知到"角色生效"）。
4. **路由守卫**：补 `meta.permissions` 校验，防直闯。
5. **用户管理界面**：`permissionOptions` 改为统一词表。
6. **后端接口鉴权**：敏感接口挂 `require_permissions`（安全兜底）。
7. **自测**：用 admin 建一个 `user` 角色账号，登录验证只能看到 7 个菜单，且直接输 `/users` 会被拦回。

## 8. 待确认的设计决策（可迭代）

- 普通用户是否需要 `tag:edit` / `message_task:edit`？（决定他能否自建任务/标签）
- 单用户 `permissions` 为空时是"用角色默认"还是"无任何权限"？（建议前者）
- 是否需要"自定义角色"（不止 admin/editor/user 三档）？当前模型仅三档，足够则不改。
- 是否要菜单分组（如"系统管理"整组对普通用户隐藏）？当前按单菜单判定即可。
